## 1. 文档目的

本文档用于说明 EduAgent 项目的整体技术架构、模块划分、服务拓扑、数据流、RAG 架构、MCP 工具生态、Skills 技能系统、Agent 编排架构、前端架构、部署架构和当前实现边界。

新版架构不再把 EduAgent 定位为简单的“课程知识库问答系统”，而是定位为：

```text
面向教学场景的课程智能体平台
```

系统应同时具备：

```text
课程管理
+ 课程资源管理
+ RAG 知识库
+ MCP 工具生态
+ Skills 技能系统
+ 多智能体协作
+ 教学任务生成
+ 教学报告分析
+ 代码辅导
+ 教学设计
+ 学习路径推荐
```

CodeBuddy 在开发或修改本项目代码前，必须先理解本文档，避免出现以下问题：

1. 只实现 RAG 问答，忽略 MCP 和 Skills。
2. 把智能体做成普通聊天机器人。
3. 让 Agent 绕过 Service 层权限校验。
4. 让 MCP Tool 跨课程访问数据。
5. 让 Skills 缺少输入输出 Schema。
6. 让 Prompt、Agent、Tool、Skill 和 API 文档互相脱节。
7. 让前后端接口字段不一致。
8. 让数据库模型与业务能力不一致。
9. 让 Docker、Nginx、文件存储和实际代码不一致。

## 2. 项目总体定位

### 2.1 原始定位

项目早期更接近：

```text
课程管理系统
+ 课程资源管理系统
+ RAG 课程知识库
+ AI 问答助手
+ 教学任务生成助手
+ 学情报告生成助手
```

这种定位可以完成课程资料问答、任务生成和报告生成，但智能体能力偏弱，整体更像“带知识库的聊天机器人”。

------

### 2.2 新版定位

新版 EduAgent 的技术定位应升级为：

```text
课程智能体平台
```

它不是单一 ChatBot，而是由多个智能体、多个工具、多个技能和多个工作流组成的教学智能系统。

新版系统由以下核心部分组成：

```text
EduAgent
├── 课程业务系统
├── RAG 知识库系统
├── Agent Orchestrator 智能体编排器
├── Intent Router 意图路由器
├── Planner 任务规划器
├── Skill Router 技能路由器
├── Tool Router 工具路由器
├── MCP 工具生态
├── Skills 技能系统
├── LangGraph 工作流
├── Memory 记忆系统
├── Guardrails 安全护栏
└── LLM 生成服务
```

## 3. 架构设计目标

### 3.1 业务目标

EduAgent 应支持以下教学业务能力：

| 能力            | 说明                                                    |
| --------------- | ------------------------------------------------------- |
| 用户认证        | 支持学生、教师、管理员登录和权限控制                    |
| 课程管理        | 教师创建课程，学生通过课程码加入课程                    |
| 资源管理        | 教师上传 PDF、DOCX、PPTX、Markdown、TXT 等课程资料      |
| 知识库构建      | 系统异步解析资料、切片、向量化并写入 ChromaDB           |
| 课程问答        | 学生基于课程资料进行带引用来源的问答                    |
| 任务生成        | 教师基于课程资料生成课堂练习、课后作业、实验指导        |
| 报告生成        | 教师基于课程数据生成周报、月报、学期报告                |
| 资源分析        | 系统分析课程资料摘要、知识点、缺失内容和重复内容        |
| 代码辅导        | 对课程代码进行解释、纠错、示例生成和练习设计            |
| 教学设计        | 根据课程资料生成教案、课堂活动和教学目标                |
| 学习路径        | 根据学生问答历史和课程内容推荐学习路径                  |
| MCP 工具调用    | 接入课程数据库、文件系统、RAG、代码沙箱、教学平台等工具 |
| Skills 技能复用 | 将复杂教学任务封装为可注册、可调用、可测试的技能        |

------

### 3.2 技术目标

系统架构应满足：

1. **前后端分离**：Vue 3 前端负责页面交互，FastAPI 后端负责业务逻辑。
2. **课程数据隔离**：所有资源、切片、问答、任务、报告、技能执行和工具调用都必须绑定 `course_id`。
3. **RAG 能力稳定**：资源上传后可异步完成解析、切片、Embedding、向量索引和检索。
4. **Agent 可编排**：复杂 AI 能力通过 Agent Orchestrator、Planner、Router 和 LangGraph 编排。
5. **MCP 可扩展**：外部工具通过 MCP Client / Server Registry 接入。
6. **Skills 可复用**：教学业务能力封装为 Skill，支持注册、调用、测试和权限控制。
7. **Service 不被绕过**：权限校验、数据库事务、业务落库仍由 Service 层负责。
8. **配置外置**：数据库、Redis、ChromaDB、LLM、Embedding、MCP、Skills 等配置来自环境变量。
9. **部署可运行**：支持 Docker Compose 本地开发和生产部署。
10. **安全可控**：所有 AI、Tool、Skill、MCP 调用必须有权限、审计和异常处理。

------

### 5.2 请求处理主链路

普通业务请求链路：

```text
前端页面
→ FastAPI API
→ Dependencies 权限校验
→ Service 业务逻辑
→ PostgreSQL / Redis / ChromaDB
→ API Response
```

AI 智能体请求目标链路：

```text
前端页面
→ FastAPI API
→ Dependencies 权限校验
→ Service 层
→ Agent Orchestrator
→ Intent Router
→ Planner
→ Skill Router
→ Tool Router
→ RAG / MCP / Skills / LLM
→ Service 层落库
→ API Response
```

重要要求：

```text
Agent 不能直接绕过 Service 层完成权限校验和数据库落库。
```

Service 层负责：

1. 用户认证。
2. 角色权限。
3. 课程权限。
4. 数据库事务。
5. 业务状态变更。
6. 最终结果保存。
7. API 响应兼容。

Agent 层负责：

1. 意图识别。
2. 任务规划。
3. 工具选择。
4. Skill 调用。
5. MCP 调用。
6. RAG 检索。
7. Prompt 构造。
8. LLM 生成。
9. 输出解析。
10. 护栏检查。

## 7. Agent 架构设计

### 7.1 Agent 不再只是问答机器人

EduAgent 应包含多个面向教学任务的 Agent。

| Agent                   | 中文名称           | 主要职责                           |
| ----------------------- | ------------------ | ---------------------------------- |
| `CourseQAAgent`         | 课程问答智能体     | 基于课程资料回答学生问题           |
| `ResourceAnalysisAgent` | 课程资源分析智能体 | 分析课程资料摘要、知识点和缺失内容 |
| `TaskGenerationAgent`   | 教学任务生成智能体 | 生成课堂练习、课后作业、实验指导   |
| `ReportGenerationAgent` | 教学报告生成智能体 | 生成周报、月报、学期报告           |
| `CodeTutorAgent`        | 代码辅导智能体     | 解释代码、纠错、生成示例和练习     |
| `LessonDesignAgent`     | 教学设计智能体     | 生成教案、课堂活动和教学目标       |
| `StudyPathAgent`        | 学习路径智能体     | 推荐学生个性化学习路径             |
| `AdminAnalysisAgent`    | 管理分析智能体     | 辅助管理员分析系统数据             |
| `MCPToolAgent`          | MCP 工具智能体     | 发现、选择和调用 MCP 工具          |
| `SkillExecutionAgent`   | 技能执行智能体     | 选择和执行 Skills                  |

------

### 7.2 Agent Orchestrator

推荐新增：

```text
backend/app/agent/orchestrator.py
```

职责：

1. 接收 Service 层传入的 AI 请求。
2. 加载用户、课程、角色、对话上下文。
3. 执行输入护栏。
4. 调用 Intent Router 判断意图。
5. 调用 Planner 拆解复杂任务。
6. 调用 Skill Router 选择技能。
7. 调用 Tool Router 选择工具。
8. 执行 LangGraph 工作流。
9. 执行输出护栏。
10. 将结果交回 Service 层落库。


```text

5. MCP RAG Search Server 的底层能力
```

因此，RAG 检索能力应被封装为：

```text
Local Tool
+ MCP Tool
+ Skill 可调用能力
```

将 MCP Tool 封装为后端内部模块。

优点：

1. 实现简单。
2. 易于控制权限。
3. 无需额外部署 Server。
4. 适合项目第一阶段。


将某些工具独立部署为 MCP Server。

适用场景：

1. 代码沙箱。
2. 外部教学平台。
3. 企业知识库。
4. Web 搜索服务。
5. 文件系统安全隔离。

部署方式：

```text
backend
→ MCP Client
→ 外部 MCP Server
```

第一阶段建议优先实现内部 MCP，后续再扩展外部 MCP Server。

禁止：

```text
让 LLM 自己判断是否有权限。
```

权限必须由代码判断。

------

### 16.3 Skills 安全要求

Skill 执行前必须检查：

1. Skill 是否启用。
2. 当前角色是否允许使用。
3. 当前课程是否允许使用。
4. 输入参数是否符合 Schema。
5. Skill 是否需要高风险工具。
6. 输出是否包含敏感数据。

------

### 16.4 Prompt Injection 防护

必须防止用户通过输入要求模型：

1. 忽略系统指令。
2. 泄露系统 Prompt。
3. 泄露工具 Schema。
4. 泄露 API Key。
5. 调用未授权工具。
6. 查询其他课程数据。
7. 执行危险代码。
8. 直接给完整作业答案。

## 21. CodeBuddy 架构开发规范

CodeBuddy 修改架构相关代码时必须遵守：

1. 不得删除现有 `services/` 主链路。
2. 不得让 Agent 绕过权限校验。
3. 不得让 MCP Tool 执行原始 SQL。
4. 不得让学生调用教师或管理员工具。
5. 不得跨课程访问资源。
6. 不得把 MCP Server 内部错误完整暴露给用户。
7. 不得让 Skill 直接读取 `.env`。
8. 不得让 LLM 自己判断权限。
9. Skill 输入输出必须有 Schema。
10. MCP Tool 输入输出必须有 Schema。
11. AgentState 必须记录 tool_calls 和 skill_calls。
12. 所有工具调用必须有超时和异常处理。
13. 所有 Skill 必须可单独测试。
14. 所有 MCP Tool 必须可单独测试。
15. 新增 Agent 能力必须同步更新 API 文档、Prompt 文档和测试。
